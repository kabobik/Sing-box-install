# План разработки GUI для sing-box

## Цель

Сделать небольшой Linux GUI для установленного `sing-box`, ориентированный на Linux Mint/Ubuntu:

- значок в системном трее с индикацией состояния сервиса;
- основное окно со статусом, кнопкой запуска/остановки и логом ошибок;
- настройки для удаленной подписки или прямого JSON-конфига;
- система профилей: сохранить конфиг под именем, выбрать активный, применить к сервису;
- автозапуск GUI при входе в систему;
- безопасное управление системным сервисом `sing-box.service`.

## Предлагаемое видение

Это должен быть не полноценный конкурент v2rayN/Nekoray, а аккуратная оболочка над уже установленным Linux-сервисом:

- GUI не запускает собственный sing-box-процесс, а управляет `systemd`-сервисом.
- Пользовательские профили хранятся в домашней папке.
- URL подписок можно хранить обычным текстом в `~/.config/sing-box-gui`.
- В `/etc/sing-box/config.json` попадает только выбранный и проверенный профиль.
- Перед применением нового конфига GUI выполняет `sing-box check`.
- Если проверка не прошла, старый рабочий конфиг не трогается.
- Перед применением профиля GUI всегда спрашивает подтверждение.

## Принятые решения

- Подписки: достаточно модели как в OpenWrt singbox-ui/HomeProxy - имя, URL, ручное обновление, позже автообновление и дополнительные параметры загрузки.
- Применение профиля: только после подтверждения пользователя.
- Хранение URL подписки: plain text в `~/.config/sing-box-gui`.
- Автозапуск GUI: нужен, включается через `.desktop` файл в `~/.config/autostart`.
- Иконка: своя "коробочка", визуально близкая по идее к sing-box, но не копия официального логотипа.

## Рекомендуемый стек

Базовая рекомендация: Python + PySide6.

Почему:

- быстро собрать MVP;
- нормальная поддержка окон, кнопок, списков, диалогов и трея;
- хорошо подходит для Linux Mint/Cinnamon;
- проще дебажить системные команды `systemctl`, `journalctl`, `sing-box check`;
- можно позже упаковать в `.deb`.

Альтернативы:

- Tauri: красивее и современнее, но больше инфраструктуры.
- Electron: быстрее по UI, но тяжелее для простой системной утилиты.
- GTK/libadwaita: нативнее для GNOME, но Mint/Cinnamon и Qt тоже чувствует себя нормально.

## Архитектура

### UI-приложение

Запускается от обычного пользователя.

Отвечает за:

- окно приложения;
- трей;
- список профилей;
- редактор/поле подписки;
- показ статуса сервиса;
- показ ошибок из журнала;
- запуск ограниченных привилегированных действий через `pkexec`.

### Хранилище профилей

Предлагаемая структура:

```text
~/.config/sing-box-gui/
  settings.json
  profiles/
    index.json
    <profile-id>/
      meta.json
      config.json
```

`meta.json`:

```json
{
  "id": "home-proxy",
  "name": "Home proxy",
  "source_type": "direct_config",
  "subscription_url": null,
  "subscription_options": {
    "auto_update": false,
    "update_interval_hours": 24,
    "allow_insecure": false,
    "use_proxy_for_update": false
  },
  "created_at": "2026-04-27T18:00:00+03:00",
  "updated_at": "2026-04-27T18:00:00+03:00",
  "last_check_status": "ok"
}
```

Варианты `source_type`:

- `direct_config` - пользователь вставляет JSON напрямую;
- `remote_config` - URL отдает готовый sing-box JSON;
- `remote_subscription` - подписка в стиле OpenWrt UI: URL обновляется вручную или по расписанию, а результат превращается в рабочий профиль.

### Подписки

Ориентир по UX: как в OpenWrt singbox-ui/HomeProxy, но без роутерной специфики.

Минимальные поля:

- имя подписки;
- URL;
- кнопка `Обновить`;
- дата последнего обновления;
- статус последнего обновления;
- текст последней ошибки.

Дополнительные поля после MVP:

- автообновление;
- интервал автообновления;
- обновлять через текущий прокси;
- разрешить insecure TLS;
- пользовательский User-Agent.

Поведение:

1. Пользователь добавляет подписку.
2. GUI скачивает данные в профиль.
3. GUI преобразует данные в `config.json` профиля, если нужен адаптер формата.
4. GUI выполняет `sing-box check`.
5. Если проверка успешна, профиль можно применить.

Первый адаптер формата:

- URL с готовым sing-box JSON.

Следующие адаптеры:

- base64/subscription с набором proxy URI;
- Clash/Mihomo YAML через отдельный конвертер;
- кастомный внешний конвертер как команда пользователя.

### Применение активного профиля

Поток:

1. Пользователь выбирает профиль.
2. GUI сохраняет изменения в пользовательский каталог.
3. GUI запускает `sing-box check -c <profile-config>`.
4. Если проверка успешна, GUI показывает диалог подтверждения применения.
5. После подтверждения GUI через привилегированный helper копирует файл в `/etc/sing-box/config.json`.
6. Helper делает резервную копию текущего `/etc/sing-box/config.json`.
7. GUI предлагает перезапустить сервис или делает это сразу, если пользователь подтвердил применение с перезапуском.

### Привилегии

Не запускать весь GUI от root.

Для системных действий использовать `pkexec` или отдельный helper с PolicyKit-правилом.

Привилегированные действия должны быть узкими:

- `deploy-config <profile-config>`;
- `start-service`;
- `stop-service`;
- `restart-service`;
- `enable-service`;
- `disable-service`.

Команды напрямую:

```bash
systemctl is-active sing-box
systemctl is-enabled sing-box
journalctl -u sing-box -p warning..alert --no-pager
sing-box check -c /path/to/profile/config.json
```

## Интерфейс

### Трей

Иконка:

- собственная иконка-коробочка, похожая по смыслу на sing-box, но не копирующая официальный логотип;
- зеленый индикатор: сервис активен;
- серый индикатор: сервис остановлен;
- красный индикатор: сервис failed;
- желтый индикатор: идет запуск, остановка, применение конфига.

Меню трея:

- Открыть;
- Старт / Стоп;
- Перезапустить;
- Активный профиль;
- Последняя ошибка;
- Настройки;
- Выход.

### Главное окно

Минимальный экран:

- крупный статус: `Работает`, `Остановлен`, `Ошибка`, `Неизвестно`;
- кнопка `Старт` или `Стоп` по текущему состоянию;
- кнопка `Перезапустить`;
- выбранный активный профиль;
- краткий блок ошибок из `journalctl`;
- кнопка перехода в настройки/профили.

### Профили

Функции:

- создать профиль;
- переименовать;
- удалить;
- дублировать;
- выбрать активным;
- проверить конфиг;
- применить;
- обновить по подписке;
- показать дату последнего обновления и последней успешной проверки.

### Настройки профиля

Режимы:

- URL удаленного sing-box JSON;
- прямой JSON-конфиг;
- подписка в стиле OpenWrt UI;
- позже: Clash/v2rayN/sing-box mixed через адаптеры или внешний конвертер.

Для direct config:

- текстовый редактор JSON;
- кнопка `Форматировать`;
- кнопка `Проверить`;
- подсветка ошибки проверки.

Для remote URL:

- поле URL;
- кнопка `Обновить`;
- кнопка `Проверить`;
- настройка автообновления;
- отображение последней ошибки загрузки.

## MVP

MVP должен закрыть самый короткий полезный путь:

1. Окно приложения.
2. Трей с индикацией `active/inactive/failed`.
3. Кнопки `Старт`, `Стоп`, `Перезапустить`.
4. Просмотр последних ошибок `journalctl -u sing-box`.
5. Создание профиля из прямого JSON.
6. Создание профиля из URL с готовым sing-box JSON.
7. Ручное обновление подписки.
8. Проверка профиля через `sing-box check`.
9. Применение профиля в `/etc/sing-box/config.json` только после подтверждения.
10. Сохранение нескольких профилей с именами.
11. Автозапуск GUI при входе в систему.

## Этапы работ

### Этап 1. Каркас

- Создать Python-проект.
- Подключить PySide6.
- Сделать главное окно.
- Сделать трей.
- Реализовать polling статуса сервиса.
- Нарисовать временную иконку-коробочку.

### Этап 2. Управление сервисом

- Обернуть `systemctl is-active`.
- Обернуть `systemctl start/stop/restart`.
- Добавить обработку ошибок прав.
- Решить, используем ли простой `pkexec systemctl ...` или отдельный helper.

### Этап 3. Логи

- Читать последние ошибки через `journalctl`.
- Показывать только полезные строки, без лишнего шума.
- Добавить кнопку обновления.
- Позже: live-tail логов.

### Этап 4. Профили

- Создать модель профиля.
- Сохранять профили в `~/.config/sing-box-gui`.
- Сделать список профилей.
- Добавить создание, переименование, удаление, дублирование.
- Добавить выбор активного профиля.

### Этап 5. Проверка и применение

- Запускать `sing-box check` для выбранного профиля.
- Показывать ошибку проверки в UI.
- Спрашивать подтверждение перед применением профиля.
- Делать резервную копию текущего `/etc/sing-box/config.json`.
- Применять профиль только после успешной проверки.
- После применения спрашивать подтверждение на перезапуск сервиса.

### Этап 6. Удаленные подписки и конфиги

- Добавить поле URL.
- Скачать URL в профиль.
- Проверить скачанный JSON.
- Показать дату последнего обновления.
- Добавить ручное обновление.
- Хранить URL plain text в `~/.config/sing-box-gui`.
- Позже: автообновление по расписанию.

### Этап 7. Упаковка

- Добавить `.desktop` файл.
- Добавить автозапуск GUI при входе в систему.
- Подготовить `.deb` или простой install script.
- Положить иконки в системные директории.

## Вопросы для обсуждения

1. GUI делаем только под Linux Mint/Cinnamon или сразу стараемся не ломать GNOME/KDE?
2. Какой набор форматов подписок нужен после первого адаптера `remote_config`?
3. Кнопка в главном окне должна быть `Старт/Стоп` одной кнопкой или две отдельные?
4. При подтверждении применения профиля объединять применение и перезапуск в один диалог или делать два шага?
5. Нужны ли параметры подписки `allow insecure`, `update через proxy`, `User-Agent` уже в MVP?
6. Нужен ли импорт текущего `/etc/sing-box/config.json` как первого профиля при первом запуске?
7. Нужен ли встроенный JSON-редактор в MVP или достаточно вставить конфиг в большое текстовое поле?
8. Нужна ли кнопка "откатить предыдущий конфиг"?
9. Нужен ли режим "только пользовательский сервис", или работаем только с системным `sing-box.service`?
10. Какой язык интерфейса: только русский или сразу рус/англ через переводы?

## Главные риски

- Права доступа к `/etc/sing-box/config.json` и `systemctl`.
- Разные desktop-окружения по-разному поддерживают tray/status notifier.
- Форматы подписок сильно отличаются; адаптеры и конвертеры могут стать отдельной большой задачей.
- Ошибка в конфиге может оставить сервис нерабочим, поэтому нужен check + backup.
- Иконка должна быть узнаваемой по идее, но не копировать официальный логотип.

## Предлагаемый первый спринт

Цель: получить запускаемый MVP с локальными профилями и простым URL-источником.

Состав:

- PySide6 приложение;
- tray icon;
- статус сервиса;
- кнопки start/stop/restart;
- окно с последними ошибками;
- локальные профили;
- URL-профиль, который скачивает готовый sing-box JSON;
- проверка и применение выбранного профиля.

После этого можно добавлять адаптеры подписок, автообновление и более тонкие параметры загрузки.
